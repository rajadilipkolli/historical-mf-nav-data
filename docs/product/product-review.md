# Product Review: Daily NAV Library

## Core Business Problem

Indian fintech organizations, independent quantitative researchers, and Java backend teams face a recurring challenge: they lack a zero-configuration, daily-updated source of historical mutual fund NAV (Net Asset Value) data. Without a reliable library, these teams are forced to build, maintain, and monitor their own complex ingestion pipelines to scrape and parse data from AMFI (Association of Mutual Funds in India). This results in duplicated effort, increased infrastructure costs, and unnecessary operational overhead.

## Product Positioning

The Daily NAV library is positioned as an **embedded, self-contained Spring Boot auto-configuration library**. 

Rather than functioning as an external service or API, it ships directly with a compressed SQLite dataset (`funds.db.zst`) of historical NAV data. This architecture ensures that there are absolutely no external service dependencies or network calls made at runtime. Once embedded, it provides blazing-fast, localized access to historical financial data.

## Explicit Product Goals & Success Outcome

*   **Primary Goal**: Eliminate the need for bespoke AMFI ingestion pipelines within Java/Spring ecosystems.
*   **Success Outcome**: Developers achieve plug-and-play access to rich NAV history via the `MutualFundService` in under 5 minutes, backed by comprehensive health and actuator observability to confidently run in production.

## Current Capabilities

The library currently offers a robust public API surface through the `MutualFundService`, featuring:

*   **NAV Lookups**: Retrieve the latest NAV, NAV by specific date, date-range history, or the last N-days history for any valid ISIN.
*   **Scheme & Fund Discovery**: Search for schemes by name, resolve specific scheme codes, and retrieve detailed fund information (combining scheme metadata and security context).
*   **ISIN ↔ Scheme Resolution**: Seamlessly map between scheme codes and their underlying plan ISINs.
*   **Analytics Integration**: Leverages pre-computed 200-DMA (Day Moving Average) analytics generated by the underlying Python ingestion pipeline.

## Prioritized Recommendations

The following near-term and strategic improvements are ranked by business value and effort to guide the product roadmap:

1.  **Return & CAGR Calculations** 
    *   *Effort*: Low | *Value*: High
    *   *Unblocks*: FinTech product teams building portfolio trackers.
    *   *Recommendation*: Implement utility functions in `MutualFundService` to calculate 1Y, 3Y, 5Y, and since-inception returns.
2.  **Additional Moving Averages (50-DMA, 100-DMA)**
    *   *Effort*: Low | *Value*: High
    *   *Unblocks*: Quants and FinTech dashboards displaying short-to-medium term trends.
    *   *Recommendation*: Expand the Python pipeline to calculate and store these additional analytics.
3.  **Holiday Calendar Awareness**
    *   *Effort*: Medium | *Value*: Medium
    *   *Unblocks*: Data Scientists needing precise rolling-window calculations.
    *   *Recommendation*: Integrate BSE/NSE holiday calendars to filter non-trading days out of analytical calculations.
4.  **Standalone Microservice / Non-Java Support**
    *   *Effort*: High | *Value*: Strategic
    *   *Unblocks*: Polyglot teams (Node.js, Go, Python backends).
    *   *Recommendation*: Wrap the library in a minimal Spring Boot application that exposes REST/GraphQL APIs, distributed as a Docker container.

## Personas

The Daily NAV library serves three primary personas, each interacting with the product through different access modes.

### 1. Java / Spring Backend Developers
*   **Access Mode**: Maven dependency + Spring Boot Auto-Configuration.
*   **Usage Patterns**: They rely on the library as an embedded database. They inject `MutualFundService` directly into their application layer to enrich transactions or display fund details.
*   **Prioritized Use Cases**:
    1.  Point-in-time NAV lookup by ISIN (High Value)
    2.  Fund metadata resolution and Scheme discovery (High Value)
    3.  Health monitoring via Actuator (Medium Value)

### 2. Quants / Data Scientists
*   **Access Mode**: Direct database download (`funds.db.zst` via GitHub Releases).
*   **Usage Patterns**: They bypass the Java layer entirely, downloading the raw SQLite database to connect it directly to Python (Pandas/Jupyter), R, or local BI tools for deep quantitative analysis.
*   **Prioritized Use Cases**:
    1.  Bulk extraction of NAV history ranges (High Value)
    2.  Market-sentiment analysis using 200-DMA (High Value)
    3.  Custom statistical modeling (Medium Value)

### 3. FinTech Product Teams
*   **Access Mode**: Indirect (via Backend Developers) or direct embedded usage.
*   **Usage Patterns**: Product Managers and frontend teams who dictate the requirements for embedding historical NAV trends, performance metrics, and scheme exploration into user-facing dashboards and mobile apps.
*   **Prioritized Use Cases**:
    1.  Displaying NAV history ranges on charts (High Value)
    2.  Scheme discovery (search by name) for user onboarding (High Value)
    3.  Displaying market-sentiment indicators (High Value)
